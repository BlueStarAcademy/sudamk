# 바둑 계가·표시 구조 정리

## 단일 소스 원칙 (계가/종료 시 보드)

- **문제**: 이미 따낸 자리에 상대 돌이 보이거나, 사석이 잘못 표시되는 등 클라이언트 보드와 서버 분석 결과가 어긋나는 현상.
- **해결**: `gameStatus === 'scoring'` 또는 `'ended'` 이고 `analysisResult`가 있을 때, 클라이언트는 **`boardState`가 아니라 `moveHistory`만 사용**해 보드를 다시 구성해 그립니다.
- **효과**: 서버가 분석한 국면과 화면에 그려지는 국면이 항상 동일해져, 따낸 돌 자리·사석·영역 마커 불일치가 사라집니다.

## 서버만 구동 vs 동시 이용자 제한

- **질문**: 클라이언트에서 대국을 구동하지 않고 서버만 쓰면 동시 이용자 제한이 커지나?
- **답**: 구조적으로는 **동시 이용자 제한이 커지지 않습니다**.
  - 이미 착수·AI 응답·계가(KataGo)는 모두 서버에서 처리되고 있습니다.
  - 클라이언트를 “표시만” 하게 바꾸면, 오히려 상태 불일치가 줄어듭니다.
  - 계가/종료 시 보드를 `moveHistory` 기반으로만 그리면, 서버가 매번 최종 보드를 푸시하지 않아도 클라이언트가 같은 최종 국면을 그릴 수 있어, 트래픽도 줄일 수 있습니다.

## 사석 표시 (정수)

- 사석 개수는 반드시 **정수**로 표시합니다.
- **UI**: `GameSummaryModal`에서 `scoreDetails.black.deadStones` / `scoreDetails.white.deadStones`는 `Math.round(Number(...))`로 표시.
- **서버**: `finalizeAnalysisResult`에서 `deadStones`를 배열 개수로 설정할 때 `Math.round(...)` 적용해 소수 값이 저장되지 않도록 함.
