# AI와의 바둑 대국 컨텐츠 처리 권장안

## 1. 현재 구조 요약

| 구분 | 내용 |
|------|------|
| **진입 경로** | 싱글플레이(스테이지), 대기실 AI 대전, 도전의 탑, 던전(월드챔피언십 등) |
| **AI 수** | 도전의 탑 20층+·전략바둑 대기실: 배포 GnuGo(GNUGO_API_URL) → 실패/타임아웃 시 goAiBot. 그 외: goAiBot(1~10단계). |
| **계가** | 서버에서 KataGo 분석 → 실패 시 수동 계가 |
| **트리거** | 클라이언트 `triggerAutoScoring` 또는 서버 `autoScoringTurns` 도달 시 |

이미 **착수·AI 응답·계가**는 모두 서버에서 처리되고 있어, “대국 구동”은 서버가 담당하는 구조에 가깝습니다.

## 2. 권장 처리 방식 (구조)

### 2.1 서버 단일 소스 유지

- **권장**: 클라이언트는 “표시 + 사용자 입력 전달”만 하고, **국면·계가·승패는 서버만** 결정하도록 유지합니다.
- **이유**  
  - 클라이언트 보드와 서버 분석이 어긋나던 문제(따낸 자리·사석 표시 등)를 이미 **moveHistory 기반 표시**로 잡았음.  
  - 대국 로직을 서버에만 두면 이런 불일치가 다시 생길 여지가 적습니다.
- **동시 이용자**: 서버만 구동해도 동시 이용자 제한이 구조적으로 커지지 않습니다. 이미 서버가 부하를 받고 있으므로, 클라이언트를 “표시만” 하게 두는 쪽이 일관성 측면에서 유리합니다.

### 2.2 계가/종료 시 보드는 moveHistory만 사용

- **이미 적용**: `gameStatus === 'scoring' | 'ended'` 이고 `analysisResult`가 있으면, 클라이언트는 **moveHistory만으로 보드를 재구성**해 그립니다.
- **권장**: 이 원칙을 유지하고, 다른 화면(리플레이·통계 등)에서도 “최종 국면”이 필요하면 **moveHistory 재생**을 단일 소스로 사용하는 것을 권장합니다.

### 2.3 AI 대국 진입 경로별 일관성

- **권장**:  
  - 모든 AI 바둑은 `isAiGame === true` 및 (필요 시) `gameCategory`로 구분.  
  - AI 수 결정은 `makeAiMove` → `makeGoAiBotMove(game, aiLevel)` 한 경로로 통일.  
  - 난이도는 `settings.aiDifficulty`(1~10) 또는 던전/스테이지별 설정에서만 읽도록 해, “어디서 시작했는지”와 무관하게 동일 규칙이 적용되게 합니다.
- **효과**: 싱글플레이/대기실/탑/던전이 같은 규칙과 같은 버그 수정 혜택을 받습니다.

## 3. AI 강도와 컨텐츠 구성

### 3.1 현재: goAiBot만 사용

- 1~10단계 프로필(따내기·영토·전투·실수율 등)으로 난이도 조절.
- KataGo는 **계가(사석·영역)** 에만 사용하고, **AI 착수에는 미사용**.
- 장점: KataGo 인프라/동시 접속에 덜 의존, 응답 시간 예측 가능.

### 3.2 선택지 A: 현재 구조 유지 (권장 기본)

- 모든 AI 대국은 goAiBot만 사용.
- “AI와의 바둑 대국” 컨텐츠는 **스테이지/탑/대기실에서 난이도(1~10)만 다르게** 제공.
- 계가는 계속 서버 KataGo(또는 수동)만 사용.
- 구현 부담 적고, 이미 정리한 moveHistory/계가 표시와 잘 맞습니다.

### 3.3 선택지 B: 강한 AI가 필요할 때 (KataGo 기반 AI)

- “프로급 AI” 같은 상위 컨텐츠만 KataGo(또는 외부 강한 엔진)로 두는 수를 두는 방식.
- **권장 구현**  
  - `settings.useKataGoAi: true`(또는 `aiTier: 'pro'`) 같은 플래그를 두고,  
  - 해당 게임만 KataGo “다음 수” API를 호출해 AI 수를 두고,  
  - 나머지 대국은 기존처럼 goAiBot만 사용.
- 주의: KataGo 동시 요청 수·타임아웃을 고려한 **큐/쓰로틀**이 있으면 안정적입니다. “동시 이용자 제한”은 이 큐/캐퍼시티 설계에 따라 결정됩니다.

### 3.4 선택지 C: 하이브리드

- 난이도 1~7: goAiBot  
- 난이도 8~10 또는 “프로 모드”: KataGo AI  
이렇게 구간을 나누면, 부하를 줄이면서 상위 구간만 KataGo에 의존할 수 있습니다.

### 3.5 클라이언트 측 AI (Electron + 로컬 GnuGo) — 전면 적용

- **설치형 앱(Electron)** 에서는 **로컬 GnuGo**로 AI 수를 계산하고, 그 수만 서버에 보냅니다.
- **설정**: AI 대국/도전의 탑 시작 시 `useClientSideAi: true` 가 설정되면, 서버는 `makeAiMove`를 호출하지 않고 클라이언트가 보낸 `PLACE_STONE`(payload에 `clientSideAiMove: true`)을 검증 후 적용합니다.
- **실행**: `npm run electron` 또는 `npm run electron:dev` (개발 시에는 `npm run start-client`로 웹 서버를 띄운 뒤 Electron 실행). GnuGo는 PATH에 있거나 `GNUGO_PATH` 환경 변수로 지정합니다.
- **효과**: 동시 100명이 둬도 서버 GnuGo 부하는 없으며, 계가는 계속 서버 KataGo로 수행됩니다.

## 4. 계가(스코어) 흐름 권장

1. **트리거**  
   - 사용자 착수 후 “계가하기” → 클라이언트가 `PLACE_STONE` + `triggerAutoScoring` 전송.  
   - 또는 `autoScoringTurns` 도달 시 서버가 자동으로 `getGameResult` 호출.
2. **서버**  
   - `gameStatus = 'scoring'` 설정 후 `getGameResult`만 호출.  
   - 보드는 가능하면 **서버의 moveHistory로 복원**한 뒤 KataGo/수동 계가에 넘겨, 클라이언트와 동일한 단일 소스 원칙 유지.
3. **클라이언트**  
   - 계가/종료 화면에서는 **moveHistory로만 보드 렌더링** (이미 적용).  
   - 사석 등 점수는 서버에서 내려준 정수 값만 표시.

이렇게 하면 “따낸 자리에 상대 돌이 나오거나 사석이 어긋나는” 문제를 구조적으로 피할 수 있습니다.

## 5. 요약 권장 사항

| 항목 | 권장 |
|------|------|
| **대국 구동** | 서버만 담당, 클라이언트는 표시·입력만 |
| **계가/종료 시 보드** | moveHistory 기반 단일 소스 유지 (현재 적용됨) |
| **AI 수** | 기본은 goAiBot(1~10) 유지; 필요 시 상위만 KataGo AI 옵션 추가 |
| **진입 경로** | 싱글/대기실/탑/던전 모두 같은 AI·계가 파이프라인 사용 |
| **동시 이용자** | 서버 단일 소스로 인한 추가 제한 없음; KataGo AI 도입 시에만 KataGo 쪽 큐/캐퍼시티 고려 |

이 방향으로 처리하면 AI 바둑 대국 컨텐츠를 일관되게 유지하면서, 이전에 겪었던 계가/표시 불일치를 재발 방지할 수 있습니다.

## 6. AI 응답 속도·동시 사용

### 6.1 “AI가 두는 속도가 느려지거나 겹치는” 문제

- **겹침(같은 게임)**  
  - **발생하지 않도록 되어 있음.** AI 턴은 **게임별**로 `aiSessionManager`에서 `isProcessing` 플래그로 한 번에 하나만 처리됩니다. 같은 게임에서 사용자 수와 AI 수가 동시에 반영되거나, AI 수가 두 번 들어가는 일은 없습니다.
- **겹침(다른 게임끼리)**  
  - **발생하지 않음.** 게임마다 세션이 분리되어 있어, A 유저 대국과 B 유저 대국의 수가 서로 섞이거나 잘못 들어가지는 않습니다.
- **느려짐**  
  - **여러 명이 동시에 사용하면** 부하에 따라 느려질 수 있습니다.  
    - 메인 서버: 게임별 lock만 있고 **전역 제한은 없어서**, 동시에 N명이 AI 대국을 하면 N개의 `makeAiMove`가 병렬로 돌 수 있습니다. 서버 CPU·메모리와 이벤트 루프 부하가 늘어나면 각 요청 처리 시간이 길어질 수 있습니다.  
    - GnuGo 사용 시: 배포한 GnuGo 서비스는 **프로세스 풀**(`GNUGO_POOL_SIZE`, 기본 4)을 사용하므로, **동시 N개 요청까지는 대기 없이** 처리됩니다. 풀보다 많은 동시 요청만 순서대로 대기합니다. 풀 크기를 늘리면 동시 사용 시에도 느려짐을 줄일 수 있습니다.

### 6.2 권장

- **goAiBot만 쓸 때**: 동시에 많은 AI 대국이 있어도 “겹침”은 없고, 서버 리소스 한도 내에서만 느려질 수 있음. 필요하면 메인 서버 스펙 업이나 수평 확장으로 완화 가능.
- **GnuGo 쓸 때**: **프로세스 풀**(`GNUGO_POOL_SIZE`, 기본 4)로 동시 N개 요청을 처리해 “국면 겹침” 없이 느려짐을 줄임. 풀보다 많은 동시 요청은 대기하므로, 트래픽이 많으면 `GNUGO_POOL_SIZE`를 늘리면 됨. 3초 타임아웃 + goAiBot 폴백으로 GnuGo 지연 시에도 게임이 끊기지 않도록 되어 있음.

## 7. GnuGo 배포(도전의 탑·전략바둑 대기실) 시 유의사항

### 7.1 배포 구조

- **도전의 탑 20층 이상**, **전략바둑 대기실 AI 대국**에서는 `GNUGO_API_URL`로 배포된 GnuGo 서비스에 HTTP 요청을 보내 수를 받습니다.
- GnuGo 서비스(Railway 등) 안에서는 **GnuGo 프로세스 풀**이 GTP 모드로 동작합니다. 요청마다 풀에서 프로세스 하나를 빌려 쓰고 반환하므로, **국면이 겹치지 않습니다**.

### 7.2 적용한 대응: 프로세스 풀 (동시 사용 시 느려짐 완화)

- **`gnugoService.ts`**에서 **프로세스 풀**을 사용합니다. 환경 변수 **`GNUGO_POOL_SIZE`**(기본값 **4**)만큼 GnuGo 프로세스를 미리 띄워 두고, 요청이 올 때마다 **빈 프로세스 하나를 할당**해 clear_board → play → genmove를 수행한 뒤 반환합니다.
- **동시에 최대 N개**(= `GNUGO_POOL_SIZE`) 요청이 동시에 처리되므로, **여러 명이 써도 풀 크기 안이면 대기 없이** 응답이 나갑니다. 풀보다 많은 동시 요청만 순서대로 대기합니다.
- **느려지지 않게 쓰려면**: 배포 환경에서 **`GNUGO_POOL_SIZE=4`**(또는 6, 8 등)로 설정해 두면 됩니다. 동시 사용자가 더 많으면 이 값을 늘리면 되고, 메모리·CPU 여유에 맞게 조정하면 됩니다.

### 7.3 그 외 문제 가능성·대응

| 가능한 문제 | 대응 |
|-------------|------|
| GnuGo 서비스 다운/네트워크 오류 | 메인 서버에서 `GNUGO_API_URL` 요청 실패 시 **goAiBot으로 자동 폴백** (이미 구현). |
| 동시 사용이 풀 크기보다 많아서 대기 | `GNUGO_POOL_SIZE`를 늘리기 (예: 8, 16). 서버 메모리·CPU 한도 내에서 설정. |
| GnuGo가 잘못된 수 반환 | `goLogic.processMove`로 유효성 검사 후, 무효면 **goAiBot으로 폴백** (이미 구현). |

### 7.4 요약

- **도전의 탑·전략바둑 대기실에서 배포한 GnuGo를 쓰는 것은 구조상 가능**합니다.
- **동시 사용 시 “겹침”**은 **요청당 프로세스 하나**를 쓰는 **프로세스 풀**로 방지했습니다.
- **느려지지 않게** 쓰려면 배포 시 **`GNUGO_POOL_SIZE`**(기본 4)를 적절히 키우면 됩니다. 풀 크기 안의 동시 요청은 대기 없이 처리됩니다.

## 8. “다른 앱처럼” 클라이언트에서 AI 두는 방식 (100명도 무리 없이)

다른 앱에서 100명이 AI 봇과 바둑을 두어도 무리 없이 쓰는 이유는, **AI 엔진을 서버가 아니라 사용자 기기(클라이언트)에서 돌리기 때문**입니다. 우리도 같은 방식으로 할 수 있습니다.

### 8.1 왜 그렇게 되나

- **서버 방식(현재)**: AI 차례마다 클라이언트가 서버에 요청 → 서버가 GnuGo/goAiBot 실행 → 결과 전송. 동시 사용자가 많으면 서버/풀 부하로 대기·느려짐이 생길 수 있음.
- **클라이언트 방식**: AI 차례를 **사용자 브라우저(또는 설치형 클라이언트)** 안에서 계산하고, 그 **수만 서버에 “AI가 둔 수”로 보냄**. 서버는 합법성만 검사하고 상태만 갱신. 서버는 GnuGo를 돌리지 않으므로 **동시 100명이 둬도 서버 부하는 거의 없음**.

즉, “클라이언트에 설치하고 두는 것처럼” 동작하려면 **AI 수를 클라이언트에서 계산**하면 됩니다.

### 8.2 우리 앱에서 가능한 구현 형태

우리 앱은 **웹**이므로, 선택지는 대략 두 가지입니다.

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **A. 브라우저에서 엔진 실행 (WASM)** | GnuGo 또는 다른 바둑 엔진을 **WebAssembly(WASM)**로 컴파일해 브라우저에서 실행. AI 차례일 때 클라이언트가 수를 계산한 뒤 서버에 전송. | 서버 부하 없음, 100명·1000명도 확장 용이. 별도 설치 없이 웹만으로 “클라이언트에서 두는” 느낌. | WASM 빌드/통합 작업 필요. 엔진에 따라 초기 로딩·메모리 사용량 이슈 있을 수 있음. |
| **B. 설치형 클라이언트(Electron 등) + 로컬 GnuGo** | PC용 앱을 Electron/TAURI 등으로 만들고, GnuGo 실행 파일을 번들. AI 차례일 때 로컬 GnuGo가 수를 두고, 그 수를 서버에 전송. | “다른 앱”처럼 완전히 로컬에서 돌리므로 서버 부하 제로. 기존 GnuGo 그대로 사용 가능. | 웹만 쓰는 사용자에게는 해당 없음. 빌드·배포·설치 경로 필요. |

둘 다 **AI 대국**에서는 “수 계산”만 클라이언트로 옮기고, **계가·승패·저장**은 지금처럼 서버가 담당하는 구조로 가져가면 됩니다.

### 8.3 서버가 할 일 (클라이언트 AI 사용 시)

- **AI 차례**: 클라이언트가 “AI가 둔 수 (x, y)”를 한 번에 보냄. 서버는 **그 수가 합법인지만 검사**하고, 합법이면 보드·moveHistory 갱신 후 다음 턴(또는 계가)으로 진행.
- **계가·종료**: 지금처럼 서버에서 KataGo(또는 수동 계가)만 수행. 클라이언트는 결과·점수만 수신.
- **부정행위**: AI 대국은 “상대가 AI”이므로, AI 쪽 수를 클라이언트가 계산하든 서버가 계산하든, 유저가 “AI 수를 조작”해도 **랭크/매너 등에 영향을 주지 않도록** 설계하면 됨. 서버는 “합법 수인가”만 검사하면 됨.

즉, **그런 방식으로 하면 됩니다.** 다만 “웹에서 그렇게 하려면” **브라우저에서 돌릴 엔진(WASM)** 이나 **설치형 클라이언트** 중 하나를 도입해야 합니다.

### 8.4 정리

- **다른 앱처럼 “클라이언트에 설치하고 두는 것처럼”** 쓰게 하려면 → **AI 수를 클라이언트에서 계산**하고, 서버는 검증·저장·계가만 하면 됨.
- **웹만 유지**하려면: WASM 바둑 엔진을 붙여서 브라우저에서 AI 수 계산 후 서버에 전송하는 방식이 필요함.
- **설치형 앱**을 만들 수 있다면: Electron 등 + 로컬 GnuGo 번들로, 지금 서버에서 하던 “AI 수 계산”만 클라이언트로 옮기면 100명이 동시에 둬도 서버는 거의 무부하로 동일하게 제공 가능함.
